What we're asking you to turn off

Three toggles

For each one: what it does, why it should be off, and what we give up in return. We're not hiding anything — every change has a cost, however small.

1

Email Address Obfuscation

Security → Settings → Email Address Obfuscation
theory · 2 min to verify
Why turn it off

This is most likely what breaks the WordPress Customizer. The feature hides email addresses from spam bots: it finds an email in the HTML and swaps it for a placeholder link. The problem is that it also reaches inside scripts — and when something in the code merely looks like an email, it injects a chunk of HTML right into the JavaScript. The script then crashes.

What we give up

Email addresses on the pages become visible to spam bots as plain text. A small thing, but we'd rather say it up front. If it matters, we can handle it on the WordPress side later.

2

Browser Integrity Check

Security → Settings → Browser Integrity Check
blocks our work
Why turn it off

It decides whether a "real browser" knocked, and turns everything else away with a 403 page. Our working tools get caught in the crossfire:

  • WordPress MCP doesn't work — the tool we use to make changes and pull pages in bulk. Right now everything has to be done by hand through a browser.
  • We can't verify the trackers remotely — meaning we cannot confirm that the CIPA work is actually done.
What we give up

Almost nothing. It's a crude header check, not real protection — the WAF and DDoS protection do the actual work, and we're leaving those alone.

If it can't be turned off: leave it on, but add a skip rule for the /wp-json/ and /wp-admin/ paths.

3

Web Analytics (RUM)

Speed → Settings → Recommendations → Web analytics
CIPA risk
Why turn it off

This is Cloudflare's own analytics tracker. It fires before the visitor clicks "Accept" — precisely what the demand letter is about. And it cannot be blocked by a consent banner (we explain why below). There's no upside either: analytics is already collected through Google Analytics.

What we give up

Traffic statistics inside the Cloudflare dashboard. The same numbers are already in GA4, so effectively nothing is lost.

4

Minor: JS Detections and Schema validation

Security → Settings
optional
Why turn it off

JS Detections injects yet another Cloudflare script into your pages — unwanted noise for the consent work. Schema validation is on but configured with nothing (zero schemas, action set to "no action") — just clutter in the settings.

What we give up

Nothing noticeable. This is housekeeping, not a fix — it can wait.

Why this matters right now

A consent banner cannot stop Cloudflare's scripts

This is the key point, and it's easy to miss. The consent plugin (Complianz) lives inside WordPress. Cloudflare adds its scripts after WordPress has already handed over the page — further downstream. The plugin physically cannot see them, and cannot block them.

1

Email Address Obfuscation

Security → Settings → Email Address Obfuscation
гипотеза · проверяется за 2 мин
Зачем выключать

Скорее всего именно он ломает кастомайзер. Функция прячет email-адреса от спам-ботов: находит почту в HTML и заменяет ссылкой-заглушкой. Проблема в том, что она лезет и внутрь скриптов — и если в коде попалось что-то похожее на почту, она втыкает туда кусок HTML. Скрипт после этого падает.

Что теряем

Email-адреса на страницах станут видны спам-ботам, как обычный текст. Мелочь, но честно предупреждаем. Если критично — потом закроем это на стороне сайта.

Browser Integrity Check

Security → Settings → Browser Integrity Check
блокирует нашу работу
Зачем выключать

Он решает, «настоящий ли браузер» постучался, и отбивает всё остальное страницей 403. Под раздачу попадают наши рабочие инструменты:

  • Не работает WordPress MCP — им мы вносим правки и выгружаем страницы пакетно. Сейчас всё приходится делать руками через браузер.
  • Нельзя проверить трекеры удалённо — то есть подтвердить, что задача по CIPA выполнена, мы не сможем.
Что теряем

Почти ничего. Это грубая проверка заголовков, а не серьёзная защита — основную работу делают WAF и DDoS-защита, их мы не трогаем.

Если выключать нельзя: оставьте включённым, но добавьте правило-исключение для адресов /wp-json/ и /wp-admin/.
